# 技术选型说明

## 翻译模型

在本次 demo 设计中，我们需要满足 离线运行、端侧可集成 这三个核心约束，因此在翻译模型（MT）的选型上，我重点评估了不同技术路线的可行性。

最终，我选择了 MarianMT / OPUS-MT 作为翻译能力的核心方案，主要基于以下几点考虑：

轻量化适合移动端：模型体积较小、推理开销低，相比大模型更适合在 iOS 端本地运行
完全离线能力：可直接加载本地模型进行推理，无需依赖网络，保证稳定性与隐私
多语言覆盖完善：OPUS 提供丰富的多语种模型，便于快速扩展翻译能力
推理稳定性高：属于确定性翻译模型，输出结果更可控，适合工具型应用场景

在方案对比阶段，我也评估了基于大语言模型（如 Gemma）的方案：

大模型：

- 模型体积大、资源消耗高，不适合移动端部署
- 推理延迟较高，实时性较差

综合来看，MarianMT 更符合当前项目的技术约束，尤其在以下方面具备优势：

- 端侧可运行（iOS / Android 均可部署）
- 工程实现简单（支持 ONNX，易集成）
- 性能与资源占用平衡良好

因此，最终选择 MarianMT / OPUS-MT 作为翻译方案，是在当前需求约束和工程可落地性之间做出的合理权衡。

## 推理引擎（Inference Engine）

在离线模型的选型上，我主要从 Hugging Face 获取模型，并重点考虑了端侧的运行方案。对比了 ONNX 和 Core ML 之后，我最终选择了 ONNX 作为当前阶段的主要推理方案，主要基于以下几点：

### 跨平台能力更强

ONNX 是一个通用的模型中间表示，支持 iOS、Android 以及服务端统一部署。这点有利于未来扩展Android，可以最大程度复用同一套模型和推理逻辑。

### 更成熟的量化与压缩能力

ONNX 在量化（如 INT8、甚至更低比特）方面生态更成熟，工具链也更完善，能够有效降低模型体积和内存占用，更适合移动端场景。

### 推理性能稳定且可控

通过 ONNX Runtime，可以灵活选择不同的执行后端（CPU / NNAPI / CoreML delegate 等），在不同设备上获得比较稳定的性能表现。

### 对Transformer支持更好

翻译模型选择了Transformer 的模型，而不是NLP模型

### 为什么没有优先选择 Core ML？

虽然 Core ML 在 iOS 端有更深度的硬件优化（如 Neural Engine），但在当前阶段存在一些实际问题：

- 对部分模型算子支持不完整，转换过程中容易踩坑
- 对动态 shape 的支持不够友好，增加适配复杂度
- 对 Transformer 类模型（尤其是复杂结构）支持还不够稳定

在项目初期，我更倾向于选择 通用性强、开发成本低、可快速验证的方案（ONNX），优先把功能跑通并保证多平台一致性。
后续在产品稳定之后，可以针对 iOS 端进行进一步优化，例如：

- 将部分模型转换为 Core ML
- 或使用 ONNX Runtime 的 Core ML delegate 做混合加速

这样可以在不影响整体架构的前提下，逐步提升性能。

## 自动语音识别(ASR) -> Whisper

在本次 demo 设计中，我们需要满足 离线运行、可控性强以及 App 内集成 这三个核心约束，因此在语音识别（ASR）方案的选型上，我重点评估了不同技术路线的可行性。
最终，我选择了 Whisper 作为语音处理的核心方案，主要基于以下几点考虑：

- 完全离线能力：Whisper 支持本地推理，无需依赖网络，能够满足离线场景的稳定性要求
- 多语言支持完善：原生支持多语种识别，适合扩展国际化能力
- 识别准确率较高：在开源 ASR 模型中表现优秀，尤其在复杂语音环境下更稳定
- 可实现准实时处理（streaming hack）：虽然不是原生流式，但可以通过分段推理实现接近实时的体验

在方案对比阶段，结合ChatGPT给出的各种ASR工具，我也评估了 Apple Speech Framework：

- 优点是系统级集成、调用简单、性能优化好
- 但存在一定限制，例如：
  - 离线能力依赖系统语音包（需要额外下载，且不可控）
  - 行为受系统策略限制，灵活性较低
  - 与 Android 平台无法复用

综合来看，Whisper 更符合当前项目的长期技术路线，尤其是在以下方面具备优势：

- 跨平台一致性（iOS / Android 可复用同一套 ASR 方案）
- 更高的可控性（模型、推理流程、优化策略均可自定义）
- 降低后续维护复杂度（避免多端维护两套 ASR 技术栈）

因此，最终选择 Whisper 作为核心 ASR 方案，是在当前需求约束和长期扩展性之间做出的权衡。

## 文字语言识别

### 常见的语言识别方案大致可以分为四类：

1. iOS 原生的 Apple NaturalLanguage、
2. Google ML Kit 的 Language Identification、
3. fastText 模型
4. CLD3

结合当前项目需求（仅对文本进行语言识别，任务相对轻量，且不涉及复杂 NLP 场景），我最终选择了 Apple NaturalLanguage，主要基于以下几点考虑：

#### 集成成本最低

- 无需引入额外模型或推理框架，直接使用系统 API 即可完成能力接入，显著降低开发复杂度与后期维护成本。

#### 性能与响应速度更优

- 作为系统级能力，调用路径更短，延迟更低，能够很好地支持实时输入等交互场景。

#### 稳定性与可靠性成熟

- 在常见语言识别任务中表现稳定，覆盖主流语言场景，能够满足 demo 及早期产品阶段的需求。

#### 更符合当前阶段的工程策略

- 在项目初期，更重要的是快速完成功能闭环和验证整体体验，而不是在非核心模块上引入额外复杂度。优先选择简单、可靠、可落地的方案，有助于提高整体开发效率。

## 文字转语音（TTS）

在文字转语音（TTS）方案上，我选择使用 iOS原生的 **AVSpeechSynthesizer**，而未引入第三方服务或本地模型，主要基于以下几点：

- 集成简单、开发成本低：系统能力开箱即用，无需接入 SDK 或处理模型推理逻辑，适合 Demo 快速落地
- 无需额外模型下载：不增加包体体积，用户体验更轻量
- 支持离线，稳定性高：无需依赖网络，在弱网/离线场景下依然可用
- 性能与功耗更优：系统级调度，延迟低、资源管理更稳定
- 多语言支持完善：内置多语言语音能力，覆盖常见使用场景
- 优先打通功能闭环：快速实现“语音识别 → 翻译 → 语音合成”链路
- 当前需求不复杂：TTS 属于辅助功能，对音色和情感要求暂时不高
- 阶段性跨平台取舍：先用 iOS 原生方案加速 Demo，后续再评估统一跨平台方案

## 解压缩工具

相比 iOS 原生的 Apple Compression framework，我最终选择了 ZIPFoundation。主要原因有几点：

1. 更贴合当前需求：我的模型压缩包本身就是 ZIP 格式，而 ZIPFoundation 对 ZIP 的读写和解压支持更直接，能够更好地匹配当前场景。
2. 易用性更好：虽然系统也有 Archive API，但相关文档和示例相对较少，实际接入成本更高；相比之下，ZIPFoundation 的 API 更清晰，使用起来更直观。
3. 技术栈一致：ZIPFoundation 本身是用 Swift 编写的，和 iOS 项目的技术栈一致，集成和维护都会更顺畅。
4. 性能足够满足需求：对于模型文件解压这种场景，ZIPFoundation 的性能已经足够，不会成为当前阶段的瓶颈。
5. 支持流式解压：它支持 streaming 解压，可以边读边解，避免一次性把整个压缩包加载进内存，更适合处理较大的模型文件，也能降低内存峰值风险。
