# 性能分析

一、测试目标

本次测试主要针对翻译模块的执行性能与结果稳定性进行验证，覆盖以下两条翻译链路：

- 中文 → 英文
  - 当前为单跳翻译链路：zho -> eng
- 中文 → 日文
  - 当前为双跳翻译链路：zho -> eng -> jpn

测试重点不在于评估模型的绝对翻译能力，而是验证在现有工程实现下：

1. 不同长度文本输入时的翻译稳定性
2. 不同链路下的执行耗时表现
3. 长时间循环调用时的内存占用情况
4. 中→日双跳链路是否存在明显语义漂移或额外性能损耗

二、测试思路

本次测试分为两部分：

- 人工测试
  - 用统一语料逐条验证翻译结果是否符合预期
  - 重点关注语义保真、数字时间地点等关键信息是否正确
- 自动测试
  - 使用同一份语料进行循环调用
  - 分组执行短文本、中等文本、长文本和混合文本
  - 记录执行耗时与 RSS 内存变化，用于性能分析

人工测试负责验证“结果对不对”，自动测试负责验证“跑得稳不稳、快不快、占多少内存”。

三、语料设计原则

人工测试使用一份独立维护的中文语料文件，所有语料均包含：

- 原文
- 英文参考结果
- 日文参考结果
- 验收关键词
- 人工判定说明

语料按长度分为三类：
短文本

- 约 8～25 个中文字符
- 单句
- 适合验证即时翻译和基础表达能力
- 覆盖内容：
  - 问候
  - 时间
  - 地点
  - 数字
  - 否定表达

中等文本

- 约 40～90 个中文字符
- 2～3 个分句
- 适合验证上下文承接和轻量逻辑关系
- 覆盖内容：
  - 计划
  - 提醒
  - 条件表达
  - 上下文衔接

长文本

- 约 140～260 个中文字符
- 段落级输入
- 适合验证长文本完整性和复杂语义保持能力
- 覆盖内容：
  - 多条件组合
  - 多动作顺序
  - 礼貌表达
  - 信息完整性

## 翻译模块性能测试

### 人工测试

人工测试主要用于验证翻译结果的可用性和语义准确性，尤其关注自动指标难以衡量的内容，包括：

- 语义是否完整
- 是否存在漏译、错译、过翻
- 时间、数字、地点、否定关系是否保持正确
- 长句是否出现截断、重复、丢条件
- 中→日双跳翻译是否出现明显中间态痕迹

### 人工验收标准

人工测试不以“逐字一致”作为判定标准，而采用可用性导向的验收规则：

```
Pass

- 语义与原文一致
- 关键信息完整
- 表达自然，无明显错误

Minor Issue

- 主体语义正确
- 仅存在轻微措辞不自然、礼貌程度偏差或局部表达生硬
- 不影响整体理解

Fail

- 核心语义错误
- 数字、时间、地点、否定关系翻译错误
- 出现明显漏译、错译、截断、乱码或语言混杂
```

### 自动化测试

自动测试主要用于验证翻译模块在连续调用场景下的性能表现，重点包括：

- 单次翻译平均耗时
- 最慢调用耗时
- 整组执行总时长
- RSS 内存起始值、结束值、峰值
- 长时间循环调用下是否存在明显内存增长或波动异常
- 不同文本长度和不同链路下的性能差异

### 自动测试原则

自动测试直接调用真实翻译服务，不使用 mock，不绕过实际路由逻辑，确保结果能够反映真实执行表现。
测试前需要先做 preflight 校验：

- 校验 中文 -> 英文 是否为单跳链路
- 校验 中文 -> 日文 是否为双跳链路
- 若所需模型未安装，则测试应立即失败，并给出明确缺失信息
- benchmark 不应将模型下载时间纳入性能统计范围

### 自动测试分组

为了更清晰地分析不同输入规模对性能的影响，自动测试按以下 4 组执行：
组 1：全短文本
仅使用短文本语料循环调用。
目的：

- 观察轻量输入场景下的平均响应速度
- 验证高频短调用下是否存在内存波动

组 2：全中等文本
仅使用中等长度语料循环调用。
目的：

- 观察日常多分句输入的稳定性与耗时变化
- 分析复杂度上升后性能变化趋势

组 3：全长文本
仅使用长文本语料循环调用。
目的：

- 验证长文本场景下的耗时与峰值内存
- 观察是否存在截断、失败率升高或性能抖动

组 4：混合文本
按照固定顺序混合短、中、长文本循环调用。
目的：

- 模拟真实用户输入长度不均匀的情况
- 观察系统在复杂负载变化下的整体表现
- 判断内存曲线是否更加波动，是否存在“长文本拖累后续请求”的现象

### 组内执行顺序

每组内固定按以下顺序执行：

1. 中文 → 英文
1. 中文 → 日文

采用固定顺序的原因是：

- 保持测试结果可重复
- 降低并发与随机调度对结果的干扰
- 方便横向比较单跳链路与双跳链路的性能差异

### 循环策略

每一组都采用固定语料和固定轮次进行串行调用，不并行执行，避免 XCTest 并发调度影响结果。
推荐按“文本越长，循环次数越少”的原则控制总测试量，避免单组测试时间过长，同时仍保留统计意义。
例如：

- 全短文本：高轮次
- 全中等文本：中轮次
- 全长文本：低轮次
- 混合文本：中轮次

### 内存采样逻辑

内存指标采用当前进程 RSS 作为观测值，测试逻辑如下：

- suite 开始前采集一次 RSS，作为起始值
- suite 执行期间按固定时间间隔轮询采样
- suite 结束后再采集一次 RSS，作为结束值
- suite 内所有采样值中的最大值作为峰值 RSS

该方式用于观察：

- 长时间执行后是否存在内存持续抬升
- 长文本或双跳链路是否导致更高峰值内存
- 不同组别之间的内存差异

#### 结果分析

本次翻译 benchmark 的第一结论是“稳定性通过，但性能分层很明显”。

从稳定性角度看：

- 560 次调用全部完成
- 没有 runtime failure
- 没有空字符串输出
- 4 组 suite 都能按预期跑完

说明当前离线翻译链路在“能否稳定执行完成”这一层面已经基本通过。

从时延表现看：

- `zh-en` 只有短句接近即时响应，平均 152ms
- `zh-en` 到中等文本后平均时延已接近 1s，长文本上升到 5.61s
- `zh-ja` 全面慢于 `zh-en`
- `zh-ja` 在 short 场景平均 1.42s，是 `zh-en` 的 9.32 倍
- `zh-ja` 在 medium、long、mixed 场景分别约为 `zh-en` 的 3.45 倍、2.37 倍、2.43 倍

这说明 `zh-ja` 的双跳链路不仅有推理次数翻倍的问题，还存在明显固定成本，短文本场景尤为突出。

从资源消耗看：

- RSS 起始值从 136.91MB 开始
- 全局峰值 RSS 达到 955.03MB
- 长文本与 mixed 场景的峰值内存都接近 1GB
- `all-long zh-en` 相比 suite 起点额外抬升约 455.80MB
- `mixed zh-ja` 即使在热态下运行，峰值仍达到 948.89MB

这说明当前模型加载与长文本解码会显著放大移动端内存压力。

从实现角度看，当前性能瓶颈主要有三类：

- `zh-ja` 走 `zh -> en -> ja` 双跳，必然增加一次完整推理
- 当前服务是单实例复用，双跳场景会放大模型切换与 warm state 管理成本
- 长文本未做更激进的增量解码优化，导致长度增长后时延快速放大

#### 指标口径说明

这份 benchmark 统计口径来自 `TranslationPerformanceTests.swift` 当前实现，重点记录：

- `averageCallDurationMs`
- `slowestCallDurationMs`
- `suiteDurationSeconds`
- `rssStartMB` / `rssEndMB` / `rssPeakMB`
- `failureCount`

需要特别说明的是，这个 benchmark 只把“报错”或“返回空字符串”视为失败，不会自动校验译文质量。因此它本质上是一份稳定性与资源报告，不是翻译质量验收报告。质量结论仍需要结合本页人工测试，以及 [模型benchmark](/link/Resource/Doc/模型benchmark.md) 一起判断。

#### 最终结论

当前翻译模块的自动化性能结论如下：

- 稳定性：通过
- 中文 → 英文：仅短文本具备较好交互性，中长文本延迟偏高
- 中文 → 日文：双跳链路性能明显偏重，不适合强实时交互
- 内存占用：峰值接近 1GB，长文本场景风险较高

如果后续继续优化这条链路，优先级建议如下：

- 引入按 `packageID` 的多模型缓存，先降低 `zh-ja` 模型切换成本
- 为长文本启用增量解码，优先压低 `all-long` 与 `mixed` 的平均时延和尾延迟
- 在 benchmark 中补充 route 切换耗时与更细粒度阶段耗时，便于归因
- 增加 P50 / P95 / P99 与更长时间 soak test，持续观察尾延迟和常驻内存

### 自动化测试

自动化测试原则  
自动化测试从 AsyncStream<[Float]> 这一层开始模拟真实用户输入，直接复用现有真实流式入口，不使用 mock，不额外实现一套识别或翻译逻辑，确保结果能够反映真实链路表现。

测试原则如下：

1. 不模拟麦克风硬件采集
2. 不模拟录音权限弹窗
3. 不引入真实 AVAudioSession
4. 只模拟“麦克风输出给识别服务的音频流”
5. 文本语料统一复用 translation_performance_corpus.json
6. 流式 case 编排、TTS 参数、chunk 策略统一由 streaming_audio_manifest.json 驱动
7. 音频在测试运行时通过系统 AVSpeechSynthesizer 生成，不预置静态音频文件
8. 生成后的音频统一转换为 16kHz、单声道、Float32
9. 音频切块后统一以 AsyncStream<[Float]> 输送到真实流式入口
10. 若 TTS 语音不可用或生成失败，当前 case 记录失败原因并继续后续 case
11. benchmark 不将模型下载或首次安装时间纳入性能统计范围

### 自动化测试前置校验

自动化测试开始前需要先做 preflight 校验：

1. 校验流式测试所需 manifest 与 corpus 均可正常加载
2. 校验中文 TTS 语音可用
3. 校验音频格式转换链路可正常输出 16kHz 单声道 Float32
4. 校验 recognition-only 直接调用真实 streamTranscription(audioStream:)
5. 校验 full-chain 直接调用真实 startLiveSpeechTranscription(...)
6. 校验中文 → 英文为当前真实单跳链路
7. 校验中文 → 日文为当前真实双跳链路
8. 若依赖模型缺失或服务不可用，测试应立即标记为失败并输出明确缺失信息

### 自动化测试分组

为了更清晰地分析不同流式输入规模和节奏对性能的影响，自动化测试按以下 4 组执行：

组 1：suite-stream-short  
仅使用 manifest 中启用的全部 short 语料。  
目的：

- 观察轻量流式输入场景下的首包速度和最终收敛速度
- 验证高频短输入下 transcript 和 translation 的稳定性
- 观察小 chunk 节奏下的 CPU 与内存波动

组 2：suite-stream-medium  
仅使用 manifest 中启用的全部 medium 语料。  
目的：

- 观察日常多分句输入的识别与翻译稳定性
- 验证 pause 变体下 provisional → stable → final 的推进过程
- 分析复杂度上升后的时延变化趋势

组 3：suite-stream-long  
仅使用 manifest 中启用的全部 long 语料。  
目的：

- 验证长音频输入场景下的最终收敛时间
- 观察长文本是否导致 revision 次数升高、timeout 增多或性能抖动
- 评估长音频下的峰值内存与峰值 CPU

组 4：suite-stream-mixed  
按固定交错顺序混合 short、medium、long 语料。  
目的：

- 模拟真实用户输入长度不均匀的流式场景
- 观察停顿、续说和 endpoint 在复杂负载下的表现
- 判断长文本是否会拖累后续 case 的延迟或资源占用
- 组内执行顺序
- 每组内固定按以下顺序执行：
  - recognition-only
  - full-chain

每种模式内固定按以下语言顺序执行：

- 中文 → 英文
- 中文 → 日文

自动化测试执行流程

- 启动 StreamingPerformanceTests.swift。
- 按固定顺序执行 suite-stream-short、suite-stream-medium、suite-stream-long、suite-stream-mixed。
- 每个 suite 从 manifest 中读取启用 case 列表，并根据 corpusID 从原 corpus 加载 sourceText、expected.en、expected.ja。
- 测试运行时使用 AVSpeechSynthesizer 基于 sourceText 生成中文语音。
- 将生成音频统一转换为线性 PCM，并进一步转换为 16kHz、单声道、Float32 样本数组。
- 由 AudioChunkReplayDriver 按 manifest 指定的 chunkDurationMs 切分为 AsyncStream<[Float]>。
- realtime 模式下按 chunk 时长真实等待，flood 模式下不等待。
- 在 recognition-only 模式中，直接调用真实 streamTranscription(audioStream:)。
- 在 full-chain 模式中，直接调用真实 startLiveSpeechTranscription(...)，由现有业务逻辑继续驱动翻译。
- 对每个 case 记录 transcript 与 translation 的各阶段事件、延迟、revision、endpoint、失败与 timeout 信息。
- case 结束时显式结束 stream，并完成当前 case 汇总。
- suite 完成后输出 suite 级统计信息，并继续执行下一组。

循环策略  
每一组都采用固定语料、固定顺序和固定轮次串行执行，不并行运行，避免 XCTest 并发调度干扰结果。推荐按“输入越长，循环次数越少”的原则控制总测试量，避免单组测试时间过长，同时保留统计意义。例如：

- short：高轮次
- medium：中轮次
- long：低轮次
- mixed：中轮次

首版默认从 15 条原始语料中启用 9 条：

- short 3 条
- medium 3 条
- long 3 条

#### 结果分析

本次流式 benchmark 的第一结论是“稳定性通过，但内容质量不过关”。

从稳定性角度看：

- 108 次调用全部完成
- 没有 failure
- 没有 timeout
- 所有 suite 都能在预期窗口内结束

说明当前流式链路在“能否完整跑通”这一层面已经具备较好的可重复性。

从资源消耗看：

- full-chain 相比 recognition-only，CPU 平均占用约提升到 2.2 倍
- full-chain 峰值 RSS 由约 793MB 提升到约 1291MB
- 长文本与 mixed 场景的峰值内存明显更高，说明翻译阶段和长时流式状态会显著放大资源压力

从链路差异看：

- `zh-ja` 全面慢于 `zh-en`
- short 场景下 `zh-ja` 端到端完成时间约 5.11s，`zh-en` 约 4.16s
- medium 场景下 `zh-ja` 约 14.30s，`zh-en` 约 13.00s
- long 场景下 `zh-ja` 约 53.57s，`zh-en` 约 51.42s

说明双跳链路确实带来了持续的额外时延与 CPU 开销，但更大的问题不是纯性能，而是质量。

#### 质量观察

本次 benchmark 虽然没有运行时失败，但 case 内容已经暴露出明显质量问题：

- 短句“我今天不去公司，改成线上。”被稳定识别为“我今天不去公司，改成县长。”
- 对应英文翻译变成 “I'm not going to the company today, but the mayor.”
- 中句中出现“行程 -> 形成”、“订位 -> 定位”、“明天下午两点 -> 明天下无两点”等明显识别偏差
- `zh-ja full-chain` 输出多次出现与原文无关的句子，整体不可用，已经不是轻微措辞问题，而是明显语义漂移或错误生成

这说明当前 benchmark 的“全绿”只能证明系统跑通，不代表结果可交付。当前自动化测试的通过标准主要基于 `failureCount` 和 `timeoutCount`，并没有把 transcript/translation 是否满足 corpus 中 `expected`、`mustPreserve` 作为失败条件，因此这份 benchmark 本质上仍是稳定性与资源报告，不是质量验收报告。

#### 指标口径说明

需要特别说明一点：`full-chain` 下的 `firstTranscriptLatencyMs` 几乎接近 0ms，这个指标当前不适合作为真实首字延迟解读。

原因是 recorder 在收到第一条 transcript state 事件时就立即记录了 `firstTranscriptTick`，即使该时刻 transcript 仍可能为空；因此：

- `firstStableTranscriptLatencyMs`
- `finalTranscriptLatencyMs`
- `firstTranslationLatencyMs`
- `finalTranslationLatencyMs`

更适合作为本次 benchmark 的有效时延指标。

#### 最终结论

当前流式语音翻译链路的结论如下：

- 稳定性：通过
- 性能：可接受但偏重，尤其是 full-chain 长文本与 mixed 场景
- 中文 → 英文：可继续优化，但已具备基础分析价值
- 中文 → 日文：当前质量不达标，暂不适合作为可用结果验收

如果后续要把这套 benchmark 用作正式质量门槛，建议下一步补上两类能力：

- 基于 corpus `mustPreserve` 的自动语义校验
- transcript 与 translation 的质量分级统计，而不只统计 failure / timeout
