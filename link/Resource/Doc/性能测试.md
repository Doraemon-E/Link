# 性能分析

## 翻译模块性能测试

### 一、测试目标

本次测试主要针对翻译模块的执行性能与结果稳定性进行验证，覆盖以下两条翻译链路：

- 中文 → 英文
  - 当前为单跳翻译链路：zho -> eng
- 中文 → 日文
  - 当前为双跳翻译链路：zho -> eng -> jpn

测试重点不在于评估模型的绝对翻译能力，而是验证在现有工程实现下：

1. 不同长度文本输入时的翻译稳定性
2. 不同链路下的执行耗时表现
3. 长时间循环调用时的内存占用情况
4. 中→日双跳链路是否存在明显语义漂移或额外性能损耗

### 二、测试思路

本次测试分为两部分：

- 人工测试
  - 用统一语料逐条验证翻译结果是否符合预期
  - 重点关注语义保真、数字时间地点等关键信息是否正确
- 自动测试
  - 使用同一份语料进行循环调用
  - 分组执行短文本、中等文本、长文本和混合文本
  - 记录执行耗时与 RSS 内存变化，用于性能分析

人工测试负责验证“结果对不对”，自动测试负责验证“跑得稳不稳、快不快、占多少内存”。

### 三、语料设计原则

人工测试使用一份独立维护的中文语料文件，所有语料均包含：

- 原文
- 英文参考结果
- 日文参考结果
- 验收关键词
- 人工判定说明

语料按长度分为三类：
短文本

- 约 8～25 个中文字符
- 单句
- 适合验证即时翻译和基础表达能力
- 覆盖内容：
  - 问候
  - 时间
  - 地点
  - 数字
  - 否定表达

中等文本

- 约 40～90 个中文字符
- 2～3 个分句
- 适合验证上下文承接和轻量逻辑关系
- 覆盖内容：
  - 计划
  - 提醒
  - 条件表达
  - 上下文衔接

长文本

- 约 140～260 个中文字符
- 段落级输入
- 适合验证长文本完整性和复杂语义保持能力
- 覆盖内容：
  - 多条件组合
  - 多动作顺序
  - 礼貌表达
  - 信息完整性

### 四、人工测试

#### 1. 测试目的

人工测试主要用于验证翻译结果的可用性和语义准确性，尤其关注自动指标难以衡量的内容，包括：

- 语义是否完整
- 是否存在漏译、错译、过翻
- 时间、数字、地点、否定关系是否保持正确
- 长句是否出现截断、重复、丢条件
- 中→日双跳翻译是否出现明显中间态痕迹

#### 2. 人工验收标准

人工测试不以“逐字一致”作为判定标准，而采用可用性导向的验收规则：

```
Pass

- 语义与原文一致
- 关键信息完整
- 表达自然，无明显错误

Minor Issue

- 主体语义正确
- 仅存在轻微措辞不自然、礼貌程度偏差或局部表达生硬
- 不影响整体理解

Fail

- 核心语义错误
- 数字、时间、地点、否定关系翻译错误
- 出现明显漏译、错译、截断、乱码或语言混杂
```

#### 3. 人工测试执行流程

人工测试按以下流程进行：

1.  在 App 内将源语言设置为“中文”
2.  目标语言切换为“英文”，逐条输入全部测试语料
3.  对照英文参考结果与验收说明，记录结果
4.  再将目标语言切换为“日文”，重复同样流程4. 每条语料记录：
    - Pass
    - Minor Issue
    - Fail

5.  对于中文→日文链路，额外记录：
    - 是否出现双跳语义漂移
    - 是否带有明显英文中间表达痕迹
    - 礼貌程度是否异常
    - 语序是否明显不自然

6.  人工测试关注重点
    - 中文 → 英文  
      重点观察：

           - 短句是否过度翻译
           - 中文省略主语时英文补全是否合理
           - 时间、地点、数字是否稳定
           - 长句是否完整，不丢条件、不重复

    - 中文 → 日文  
      除上述项外，额外关注：

          - 双跳链路是否带来语义偏移
          - 是否出现英文式表达残留
          - 是否存在日语表达不自然的问题
          - 礼貌程度是否与原句场景匹配

### 五、自动测试逻辑

#### 1. 测试目的

自动测试主要用于验证翻译模块在连续调用场景下的性能表现，重点包括：

- 单次翻译平均耗时
- 最慢调用耗时
- 整组执行总时长
- RSS 内存起始值、结束值、峰值
- 长时间循环调用下是否存在明显内存增长或波动异常
- 不同文本长度和不同链路下的性能差异

#### 2. 自动测试原则

自动测试直接调用真实翻译服务，不使用 mock，不绕过实际路由逻辑，确保结果能够反映真实执行表现。
测试前需要先做 preflight 校验：

- 校验 中文 -> 英文 是否为单跳链路
- 校验 中文 -> 日文 是否为双跳链路
- 若所需模型未安装，则测试应立即失败，并给出明确缺失信息
- benchmark 不应将模型下载时间纳入性能统计范围

#### 3. 自动测试分组

为了更清晰地分析不同输入规模对性能的影响，自动测试按以下 4 组执行：
组 1：全短文本
仅使用短文本语料循环调用。
目的：

- 观察轻量输入场景下的平均响应速度
- 验证高频短调用下是否存在内存波动

组 2：全中等文本
仅使用中等长度语料循环调用。
目的：

- 观察日常多分句输入的稳定性与耗时变化
- 分析复杂度上升后性能变化趋势

组 3：全长文本
仅使用长文本语料循环调用。
目的：

- 验证长文本场景下的耗时与峰值内存
- 观察是否存在截断、失败率升高或性能抖动

组 4：混合文本
按照固定顺序混合短、中、长文本循环调用。
目的：

- 模拟真实用户输入长度不均匀的情况
- 观察系统在复杂负载变化下的整体表现
- 判断内存曲线是否更加波动，是否存在“长文本拖累后续请求”的现象

#### 4. 组内执行顺序

每组内固定按以下顺序执行：

1. 中文 → 英文
1. 中文 → 日文

采用固定顺序的原因是：

- 保持测试结果可重复
- 降低并发与随机调度对结果的干扰
- 方便横向比较单跳链路与双跳链路的性能差异

#### 5. 循环策略

每一组都采用固定语料和固定轮次进行串行调用，不并行执行，避免 XCTest 并发调度影响结果。
推荐按“文本越长，循环次数越少”的原则控制总测试量，避免单组测试时间过长，同时仍保留统计意义。
例如：

- 全短文本：高轮次
- 全中等文本：中轮次
- 全长文本：低轮次
- 混合文本：中轮次

#### 6. 采集指标

自动测试采集两层指标：

全局 run 级指标  
表示整次 benchmark 的整体表现：

- runStartAt
- runEndAt
- totalDurationSeconds

suite 级指标  
表示每个“文本分组 + 翻译链路”的表现：

- suiteStartAt
- suiteEndAt
- suiteDurationSeconds
- callCount
- averageCallDurationMs
- slowestCallDurationMs
- rssStartMB
- rssEndMB
- rssPeakMB
- failureCount

#### 7. 内存采样逻辑

内存指标采用当前进程 RSS 作为观测值，测试逻辑如下：

- suite 开始前采集一次 RSS，作为起始值
- suite 执行期间按固定时间间隔轮询采样
- suite 结束后再采集一次 RSS，作为结束值
- suite 内所有采样值中的最大值作为峰值 RSS

该方式用于观察：

- 长时间执行后是否存在内存持续抬升
- 长文本或双跳链路是否导致更高峰值内存
- 不同组别之间的内存差异

#### 8. 失败处理逻辑

自动测试不因单条翻译失败而立即中断整个 suite。
若某次调用失败，应：

- 记录失败 case 的 id
- 记录失败发生的链路与分组
- 继续执行剩余样本
- 在 suite 汇总中统计 failureCount

这样可以避免因单个异常样本影响整组数据，并帮助后续定位特定失败模式。9. 日志输出逻辑
自动测试输出两类日志：
控制台摘要
用于开发阶段快速查看结果，内容包括：

- benchmark 开始与结束时间
- 每组测试开始与结束时间
- 平均耗时
- 最慢耗时
- RSS 起始 / 结束 / 峰值
- 失败数量

结构化 JSON
用于后续做横向对比、历史归档和趋势分析，内容包括：

- 全局 run 信息
- 每个 suite 的详细统计
- 失败 case 列表
- 输出文件路径

### 六、性能分析维度

本次性能测试的分析重点主要围绕以下几个维度展开：

#### 1. 链路复杂度差异

比较：

- 中文 → 英文
- 中文 → 日文

重点分析双跳链路是否显著增加：

- 平均耗时
- 最慢耗时
- 峰值内存
- 失败率

#### 2. 文本长度差异

比较：

- 短文本
- 中等文本
- 长文本

重点分析随着输入长度增加：

- 耗时是否线性增长
- 峰值内存是否明显抬升
- 长文本是否更容易失败或出现结果异常

#### 3. 稳定性差异

重点观察：

- 同一组内各次调用耗时是否波动过大
- 混合场景下是否比纯长度场景更不稳定
- 长时间循环后 RSS 是否持续增长

#### 4. 结果与性能的平衡

对于中→日双跳链路，需要综合人工测试结果和自动测试结果判断：

- 性能是否可接受
- 结果质量是否足够稳定
- 是否值得继续保持双跳策略
- 是否有必要后续引入直连模型优化

#### 结果分析

本次翻译性能测试运行于 2026-04-08 05:51:39Z 至 2026-04-08 06:22:23Z，总耗时 1843.80s，共完成 560 次翻译调用，0 失败。整体功能稳定，但性能分层明显：zh-en 路由表现良好，zh-ja 路由存在显著时延放大；长文本与混合负载场景下，延迟和内存占用均明显上升。

结合测试实现 TranslationPerformanceTests.swift 与翻译服务实现 MarianTranslationService.swift，当前性能瓶颈主要集中在多跳路由的模型切换成本，以及长文本解码阶段未启用缓存优化。

性能分析

zh-ja 路由性能显著偏低
zh-ja 并非直连模型，而是经由英文中转，理论上双跳会带来约 2 倍左右推理成本，但当前实测短文本达到 9.3x 差距，说明瓶颈不只来自推理次数翻倍。

单实例单模型缓存导致频繁模型切换
当前服务只维护一个 loadedState，见 MarianTranslationService.swift (line 34)。当 zh-ja 执行两段翻译时，会在 zh-en 与 en-ja 模型之间反复切换，loadState 会重建 tokenizer 和 ORT session，见 MarianTranslationService.swift (line 262)。
这会显著放大短文本场景的固定成本，是当前最主要的性能损失来源。

长文本解码成本增长过快
当前实现未启用 decoder_with_past_model.onnx。
在生成阶段，decoder 每一步都重新输入完整历史 token 序列，因此文本越长，计算量增长越明显，直接导致 all-long 与 mixed 场景时延抬升。

内存占用接近高水位
RSS 从 136.9MB 起步，峰值接近 955MB。内存采样实现见 ProcessMemorySampler.swift。
由于整个 benchmark 复用同一个 MarianTranslationService 实例，后续 suite 的 rssStartMB 会继承前序状态，因此这更像“常驻内存逐步累积”而不是单 suite 独立占用。虽然当前不能直接判定泄漏，但对移动端或低内存设备而言已构成风险。

优化项

引入按 packageID 的多模型缓存
将当前单一 loadedState 改为多模型缓存池，至少同时缓存 zh-en 和 en-ja 两个已加载 session。
预期收益：显著降低 zh-ja 的固定切换成本，优先改善 all-short 与 mixed 场景。

启用 decoder_with_past 做增量解码
避免每个 token 生成时重复输入完整历史序列。
预期收益：显著降低 all-long、mixed 的平均延迟与尾延迟。

为 route/模型切换增加独立计时
在 benchmark 中单独记录 loadState 耗时、encoder 耗时、decoder 耗时。
预期收益：将“模型加载慢”与“推理慢”拆分，便于后续优化归因。

区分冷启动与热启动测试
