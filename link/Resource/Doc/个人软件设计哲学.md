# 个人软件设计哲学

我相信，好的软件不应该喧闹。  
它应该安静地存在，在需要的时候，刚好出现。

软件存在的意义，是解决真实的问题，而不是制造更多选择。  
不追求复杂，不炫技，只专注把一件事做到足够好。

流程应当自然流动，界面应当克制简洁。  
用户不需要学习，也不需要被引导。  
直觉，就是最好的交互。

我倾向于构建轻量的产品。  
能用系统能力解决的，就不引入额外依赖。  
减少体积，也减少不确定性，让软件更稳定、更长久。

我尊重设备资源。  
电量、CPU、内存，都是用户的，而不是应用可以随意消耗的。  
软件应该节制，而不是贪婪。

我也尊重用户本身。  
没有广告，不强制登录。  
尽可能保护隐私，让数据属于用户，而不是平台。

我不试图做一个“什么都能做”的产品。  
我只做一件事，并把它做到足够好。

清晰的边界，是产品的一部分。  
不是所有功能都值得被实现。

差异，不来自功能的堆叠，  
而来自取舍。

## Firenote · 一个克制的产品实践

### 产品定位

Firenote 是一款围绕 “分红收入” 设计的投资记录工具。  
它不试图覆盖完整的投资流程，而是专注于一个问题：  
如何更清晰地理解和管理自己的被动收入。

目标用户是：  
关注长期投资、希望通过分红获得稳定现金流的人。

### 为什么只做“分红”

在设计之初，我刻意放弃了很多“看起来合理”的功能：

- K线与技术指标
- 实时交易数据
- 高频操作与短期收益分析

原因很简单：
这些功能并不服务于“被动收入”这个核心目标。  
我希望用户打开应用时，关注的是：

- 我每年能拿到多少分红
- 我的现金流是否在增长

而不是被短期波动所干扰。

### 核心设计：让关键信息直接出现

在信息设计上，我优先保证两点：

- 核心指标直接可见
- 不依赖层层点击获取信息

例如：

- 首页直接展示「年分红」与「收益率」，让用户了解当前年的收入情况
- 首页直接展示整体资产的平均股息率，让用户知道自己的现金流增长速度
- 首页直接展示累计分红统计，让用户了解自己收到了多少钱

这些设计的目标是：  
用户打开应用的一瞬间，就能知道“现在的状态”。

### 轻量化实现：优先使用系统能力

在技术实现上，我尽量避免引入不必要的依赖：

- 数据同步 → iCloud / CloudKit
- 本地存储 → SwiftData
- UI 构建 → SwiftUI

这样做带来的好处：

- 应用体积更小
- 系统稳定性更高
- 同步体验更自然（无需额外账号体系）

同时也降低了长期维护成本。

### 性能与资源控制

Firenote 的使用场景是“低频但长期”，
因此我特别关注资源使用的克制：

- 避免不必要的计算与刷新
- 控制数据更新频率
- 减少后台任务

目标不是“更快”，而是：
在长期使用中保持稳定和安静。

### 用户体验原则

在产品策略上，我坚持：

- 不强制登录
- 不展示广告
- 尽量本地化与用户可控的数据
- 普通用户和付费用户体验一致，只在资产管理的数量上进行限制

例如：

- 用户数据通过 iCloud 同步，而不是上传到自建服务器，这样可以降低使用门槛，同时增强用户信任。
- 普通用户只能添加两个股票资产，但是能使用全部完整功能
- 用户打开App就能直接使用，不强制必须注册和登录

### 边界与取舍

Firenote 并不是一个“完整的投资工具”。
它明确选择了：

- 长期
- 分红
- 被动收入

并主动放弃：

- 短期交易
- 高频决策工具
- 复杂分析体系

我认为：
产品的边界，本身就是设计的一部分。

### 总结

Firenote 是一次关于“克制”的实践：

- 在功能上做减法
- 在体验上做简化
- 在技术上保持轻量

它不是为了满足所有人，而是为了更好地服务一类人。
