# 架构设计说明

## 架构设计步骤

- 分析需求
- 提炼需求点
- 根据需求点去设计需要的功能点
- 根据功能点去寻找可行的解决方案
- 根据解决方案进行技术选型
- 根据选中的工具来封装service
- 根据需求点进行业务流程设计

## 架构设计目标

- 支持文本输入翻译与语音输入翻译两条主链路。
- 优先使用本地模型完成推理，降低对在线服务的依赖。
- 将 UI、业务编排、底层能力、持久化分层，降低耦合。
- 支持翻译模型和语音模型的下载、安装、恢复与删除。
- 支持流式输出，提升交互实时性。

## 总体架构

- 表现层：SwiftUI 页面与组件，负责状态绑定和界面渲染。
- 应用层：HomeStore + Workflow，负责状态管理与业务流程编排。
- 能力层：语言识别、翻译、语音识别、TTS、音频播放、模型资产管理。
- 数据层：SwiftData、UserDefaults、本地模型文件与资源目录。

## 分层设计

### 表现层

入口页面是 HomeView.swift，负责：

- 绑定 HomeStore 生成的视图状态
- 展示消息列表、输入栏、工具栏、下载管理页
- 响应用户操作并转发给 Store

下载管理页面在 ModelAssetsView.swift，只负责资产状态展示，不承载下载核心逻辑。

### 应用层

HomeStore 是 Home 场景的状态中心，见 HomeStore.swift。
职责包括：

- 维护页面状态、会话状态、流式翻译状态、录音状态、下载状态
- 聚合多个 workflow
- 将底层能力转换成页面可消费的 ViewState

Workflow 负责拆分业务流程：

- HomeMessageWorkflow：文本输入翻译
- HomeSpeechWorkflow：语音录音、实时转写与翻译
- HomeMessageLanguageWorkflow：消息语言切换与重翻译
- HomeDownloadWorkflow：模型下载提示与下载管理
- HomePlaybackController：TTS 和录音文件播放控制

这种设计避免了 HomeStore 过度膨胀，保留了“单状态中心 + 多流程模块”的结构。

### 能力层

能力层位于 Core/Service，核心服务如下：

- 文本语言识别：SystemTextLanguageRecognitionService
  - 基于系统 NaturalLanguage 框架识别输入语言。
- 翻译服务：MarianTranslationService
  - 基于 Marian 模型和 ONNX Runtime 本地推理，支持直译和经英语中转的两段式翻译。
- 语音识别服务：WhisperSpeechRecognitionService
  - 基于 Whisper 本地识别，支持完整转写与流式转写。
- 文本转语音：SystemTextToSpeechService
- 音频文件播放：SystemAudioFilePlaybackService
- 模型资产服务：ModelAssetService
  - 统一管理翻译模型和语音模型的下载、恢复、安装、删除与快照广播。

### 数据层

- 会话与消息：SwiftData
  - 模型定义见 ChatSession.swift 和 ChatMessage.swift
- 轻量设置：AppSettings
  - 基于 UserDefaults
- 会话仓储：HomeSessionRepository
  - 负责 session/message 的创建、删除、查询与本地音频文件清理
- 模型资源管理：
  - 保存在本地目录，安装与索引由 TranslationModelPackageManager、-SpeechModelPackageManager 负责

## 启动与依赖注入

应用启动入口在 linkApp.swift。
设计方式为手写依赖装配：

- 启动时统一创建 catalog repository、package manager、asset service、translation/speech/TTS/playback 等服务
- 聚合进 HomeDependencies
- 注入 ContentView -> HomeView -> HomeStore

## 核心业务流程

### 文本翻译流程

#### 一句话概括

文本输入 -> 语言识别 -> 模型检查 -> 消息落库 -> 一次性完成翻译 -> 通过 TypingRenderer 做伪流式展示 -> 写回最终译文。

![文本翻译流程](./images/flow.png)

#### 具体步骤

1. 文本输入 -> 点击发送 -> HomeView 调用 HomeStore.sendCurrentMessage(in:)。
2. HomeStore 将请求转发给 HomeMessageWorkflow.sendCurrentMessage(in:)，自身只作为状态入口，不直接处理发送逻辑。
3. HomeMessageWorkflow 先清洗输入文本，并校验当前系统不处于录音中、语音转写中或语音模型安装中。
4. 文本语言识别 -> 调用文本语言识别服务识别源语言 -> 结合当前选中的目标语言组成翻译语言对。
5. 模型检查 -> 检查该语言对所需翻译模型是否已安装并可用。
6. 若模型未准备好 -> 弹出下载提示 -> 本次发送终止。
7. 若模型可用 -> 创建或复用当前会话 -> 写入新的 ChatMessage -> 原文先落库，译文暂为空。
8. 消息写入后 -> 清空输入框 -> 在 HomeStore 中建立翻译中的临时展示状态。
9. 工作流启动翻译任务 -> ConversationStreamingCoordinator 调用底层 TranslationService 执行翻译。

10. TranslationService 先一次性生成完整译文。
11. 完整译文生成后 -> 交给 TypingRenderer 拆分为多个增量片段 -> 以伪流式方式逐步输出。
12. HomeMessageWorkflow 持续消费这些增量事件，并把中间展示结果回写到 HomeStore。
13. HomeView 订阅 HomeStore 状态，将这些中间结果实时渲染到消息列表中。
14. 展示完成后 -> 将最终译文写回对应的 ChatMessage -> 清理临时展示状态 -> 消息进入稳定展示状态。
15. 若翻译过程中发生异常 -> 将失败信息写入消息，并同步更新界面状态。

### 语音翻译流程

1. 语音输入分为两种模式：普通语音模式和沉浸式语音模式。
2. 两种模式共享同一段前置链路：点击语音入口 -> 检查 Whisper 模型 -> 开始录音 -> 实时转写 -> 停止录音后生成最终 transcript。
3. 两种模式的差异主要出现在“最终 transcript 生成之后”的翻译处理阶段。

![语音翻译流程](./images/flow2.png)

#### 普通语音模式

#### 一句话概括

普通语音模式：录音 -> 实时转写 -> 停止后生成 transcript -> 交给 HomeMessageWorkflow 做标准伪流式翻译 -> 写回 speech message。

#### 具体步骤

1. 点击普通语音按钮 -> HomeSpeechWorkflow 启动普通语音采集流程。
2. 模型检查 -> 检查 Whisper 模型是否可用；若未安装，则进入语音模型下载提示流程。
3. 开始录音 -> MicrophoneRecordingService 启动录音并输出音频流。
4. 实时转写 -> ConversationStreamingCoordinator 驱动实时语音转写流 -> WhisperSpeechRecognitionService 持续输出稳定文本和临时文本。
5. 系统将实时转写结果持续回写到当前 speech message 的临时状态中，用于界面展示。
6. 停止录音 -> 生成最终 transcript -> 确定本次语音消息的源语言。
7. transcript 落库 -> 将最终 transcript 和录音文件引用写回当前 speech message。
8. 翻译模型检查 -> 检查源语言到目标语言所需的翻译模型是否可用。
9. 若翻译模型未准备好 -> 提示下载模型 -> 将提示信息写入当前 speech message。

10. 若翻译模型可用 -> 将最终 transcript 交给 HomeMessageWorkflow.startSpeechTranslation(...)。
11. HomeMessageWorkflow 复用标准文本翻译链路执行翻译。
12. 底层翻译依然是一次性完成整句翻译，再通过 TypingRenderer 做伪流式展示。
13. 展示完成后，将最终译文写回当前 speech message。
14. 最终形成一条包含录音、transcript 和译文的 speech message。

#### 沉浸式语音模式

#### 一句话概括

沉浸式语音模式：录音 -> 实时转写 -> 录音过程中生成沉浸式预览 -> 停止后生成 transcript -> 由 HomeSpeechWorkflow 完成最终翻译收敛 -> 写回 speech message。

#### 具体步骤

1. 点击沉浸式语音按钮 -> HomeSpeechWorkflow 启动沉浸式语音流程。
2. 模型检查 -> 检查 Whisper 模型是否可用；若未安装，则进入语音模型下载提示流程。
3. 开始录音 -> MicrophoneRecordingService 启动录音并输出音频流。
4. 实时转写 -> ConversationStreamingCoordinator 驱动实时语音转写流 -> WhisperSpeechRecognitionService 持续输出稳定文本和临时文本。
5. 在录音过程中，HomeSpeechWorkflow 会基于稳定转写片段生成沉浸式预览状态，并尝试对稳定片段做预翻译展示。
6. 停止录音 -> 生成最终 transcript -> 确定本次语音消息的源语言。
7. transcript 落库 -> 将最终 transcript 和录音文件引用写回当前 speech message。
8. 此时不会再走普通模式中的 HomeMessageWorkflow.startSpeechTranslation(...)。
9. HomeSpeechWorkflow 会直接基于最终 transcript 和分段结果执行沉浸式最终翻译收敛，以长的为准。

10. 最终翻译结果写回当前 speech message，并结束沉浸式展示状态。
11. 最终形成一条包含录音、transcript 和沉浸式翻译结果的 speech message。

### 模型下载设计

- ModelAssetService 作为统一入口，同时管理 translation asset 和 speech asset。
- ResumableAssetArchiveService 支持断点续传与下载状态恢复。
- ModelAssetSnapshotStore 用 AsyncStream 向 UI 广播快照。
- 下载页只订阅快照并展示，不直接处理下载细节。

### 可优化点

- HomeStore 仍然偏重，后续可继续下沉部分状态到子模块
- Home feature 目前承担了较多业务入口，可逐步抽出更清晰的 domain boundary
